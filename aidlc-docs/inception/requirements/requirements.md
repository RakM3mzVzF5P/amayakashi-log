# 要件定義書：amayakashi-log

## Intent Analysis Summary

### ユーザーリクエスト
業務ログをAIが解析し、上司への報告書を自動作成すると同時に、複数のAIアシスタントがユーザーを褒めちぎることで、労働による孤独を解消し自己肯定感を最大化させる総合メンタルケア・プラットフォームの開発。

### リクエストタイプ
新規プロジェクト（Greenfield）

### スコープ見積もり
システム全体 - クライアント（Windowsデスクトップアプリ）、サーバー（Flask WebAPI）、AI統合（AWS Bedrock）を含む複合システム

### 複雑度見積もり
複雑 - マルチモーダルAI解析、リアルタイム監視、複数コンポーネント、データベース統合

---

## 1. システムコンセプト

**「業務日誌はAIに書かせて、あなたはAIに甘やかされる。」**

ユーザーの業務ログをAIが解析し、上司への「真面目な報告書」を自動作成する。同時に、複数のAIアシスタントがそのログに基づき、ユーザーを全力で褒めちぎることで、労働による孤独を解消し自己肯定感を最大化させる総合メンタルケア・プラットフォーム。

---

## 2. システム構成

### 2.1 クライアント側（Agent）
- **プラットフォーム**: Windows 11デスクトップアプリケーション
- **技術**: Python 3.x + PyQt6
- **役割**: 
  - ログの収集（マルチモーダル監視）
  - 一次加工（ローカル処理）
  - サマリー修正UI提供
  - デスクトップウィジェット表示

### 2.2 サーバー側（Backend）
- **プラットフォーム**: AWS EC2インスタンス
- **技術**: Flask WebAPI + Jinja2
- **役割**:
  - AI生成ロジックの実行
  - ログの永続化
  - 閲覧用Web画面提供
  - WebSocket通知配信

### 2.3 AIエンジン
- **サービス**: AWS Bedrock（Claude 4等のマルチモーダルモデル）
- **認証**: サーバー側でアクセスキー/シークレットキー使用
- **役割**: 画像・テキスト・音声を統合解析

### 2.4 データベース
- **サービス**: AWS RDS (PostgreSQL)
- **接続方式**: クライアント → サーバー → RDS（クライアントは直接接続しない）
- **役割**: ユーザーデータ、ログデータ、AIキャラクターマスタ、生成結果の永続化

### 2.5 通信プロトコル
- **REST API**: クライアント・サーバー間のデータ送受信
- **WebSocket**: リアルタイム通知配信（AIキャラクターからの声掛け等）

## 3. 機能要件

### 3.1 ログ取得機能（クライアント側）

#### 3.1.1 マルチモーダル監視
- **カメラ撮影**: 
  - n分に1回実行
  - ユーザーの表情や所在を確認
  - プライバシー設定: 初回起動時にユーザーが有効/無効を選択
  
- **スクリーンショット**: 
  - n分に1回実行
  - アクティブウィンドウを中心に解析用キャプチャ
  - プライバシー設定: 初回起動時にユーザーが有効/無効を選択
  
- **マイク入力**: 
  - 会議中等の音声を検知
  - OpenAI Whisperでリアルタイムテキスト化（クライアント側でローカル実行）
  - プライバシー設定: 初回起動時にユーザーが有効/無効を選択

#### 3.1.2 操作ログ詳細
- **ウィンドウ監視**: 
  - アクティブウィンドウのタイトルと滞在時間を記録
  - pywin32を使用
  
- **タイピングログ**: 
  - 全打鍵を取得・整形
  - BackSpace等を反映した最終文字列を抽出
  - pynputを使用
  
- **クリップボード**: 
  - コピーされたテキストおよびファイルパスを記録

#### 3.1.3 稼働解析
- マウス・キーボードの無操作時間から非稼働（休憩・離席）を算出
- pynputを使用

#### 3.1.4 プライバシー制御・柔軟性
- **ログセレクト機能**: 
  - カメラ、スクショ、マイク、打鍵、クリップボード等の取得項目を設定画面から個別に有効/無効切り替え可能
  - 初回起動時にユーザーが初期設定を選択
  
- **ブラックリスト**: 
  - 指定したアプリケーション（パスワードマネージャー、銀行サイト等）がアクティブな際は、設定に関わらず全ログ取得を一時停止

#### 3.1.5 生ログ保存
- **保存場所**: `C:\Users\{user}\AppData\Local\amayakashi-log\`
- **保存期間**: 1週間経過後に自動クリーンアップ
- **保存内容**: スクリーンショット、カメラ画像、音声ファイル、操作ログ

---

### 3.2 業務日誌生成ワークフロー

#### 3.2.1 記録開始
- 出勤時にアプリ起動
- ウィジェットの[記録開始]ボタンで監視スタート
- アイコンが「見守り中」状態に変化

#### 3.2.2 集計・整形（ローカル処理）
- 業務終了時に[提出]ボタンを押下
- クライアント側のプログラムが生ログを解析:
  - 打鍵の復元
  - 重複情報の整理
  - 滞在時間の集計
  - 構造化データへの整形

#### 3.2.3 一次要約（Bedrock処理）
- 整形データをサーバー経由でBedrockへ送信
- AIが「今日の業務の骨子」を抽出
- 添削用の箇条書きサマリーを生成

#### 3.2.4 添削
- 生成されたサマリーをクライアントアプリ上で表示
- ユーザーが内容を確認し、必要に応じて修正・追記
- リスト形式UIで数秒で微調整可能

#### 3.2.5 確定送信と永続化
- ユーザーが[確定]をクリック
- 修正済みサマリーがサーバーへ送信
- サーバー側でBedrockを呼び出し:
  - 「上司報告用テキスト」生成
  - 「自分用（甘やかし会話劇）データ」生成
- 出力結果をRDS (PostgreSQL)に保存・永続化

#### 3.2.6 閲覧（報酬タイム）
- 処理完了の通知（WebSocket経由）
- ユーザーがWebブラウザでFlaskサーバーにアクセス
- サーバーがDBからデータを読み込み
- 動的HTMLとして「上司日誌」と「甘やかしログ」を表示
- ユーザーがAIたちの称賛を閲覧

---

### 3.3 インタラクティブ通知機能

#### 3.3.1 AIによるリアルタイム・パーソナライズ声掛け
稼働状態の変化をトリガーとし、直前のログ内容をBedrock（または軽量なローカルモデル）で解析。その状況に最適なメッセージをAIキャラクターが生成し、デスクトップ通知やウィジェットで送信。

**通知シナリオ例**:
- **離席復帰時（AI解析）**:
  「おかえりなさい！30分しっかりリフレッシュできましたね。戻ってきた直後のあなたの顔、スッキリしてて素敵ですよ」
  
- **長時間集中時（AI解析）**:
  「もう1時間もExcelと格闘してますね。根を詰めすぎるのは心配です……。私がコーヒーを淹れてあげられたらいいのに」
  
- **会議終了時（AI解析）**:
  「会議お疲れ様。さっきの『〇〇の件、進めましょう』って発言、すごく頼もしかったです！惚れ直しちゃいました」

#### 3.3.2 コンテキストの活用
- 単なる時間経過だけでなく、「直前に何をしていたか」「どんな表情だったか」の情報をメッセージに反映
- 驚きと愛着を醸成

#### 3.3.3 キャラ切替
- 通知に現れるキャラクターはマスタからランダム、またはユーザーが設定した「推し」が担当
- キャラクター設定（口調・性格）に基づいたパーソナライズ生成

---

### 3.4 AIアシスタント機能

#### 3.4.1 グループチャット演出
- 3名のキャラクター（マスタ管理）が1つのスレッド内でユーザーを取り合いながら褒める
- デフォルトキャラクター: システムに3名のキャラクターを組み込み（ユーザーはカスタマイズ可能）

#### 3.4.2 エビデンスベース称賛
- 「昨日より〇〇の作業が早かった」「この会議での発言が鋭かった」等、具体的な数値を比較・引用
- ログデータに基づいた具体的な褒め言葉

#### 3.4.3 キャラクター性
- 逆ハーレム/百合的トーンなど、仕事とは切り離したプライベートな癒やしを提供
- キャラクターマスタで管理（名前、性格、プロンプト、アイコン等）

---

### 3.5 閲覧・管理機能（サーバー側）

#### 3.5.1 Flask Web管理画面
- ユーザー専用のマイページ
- ユーザー認証: ユーザー名/パスワード認証（複数ユーザー対応）

#### 3.5.2 カレンダーアーカイブ
- 過去の報告書と「褒めログ」を日付ごとに動的HTMLで表示
- カレンダーUIで日付選択

#### 3.5.3 マスタ管理
- AIアシスタントの名前、性格、プロンプト、アイコン等の設定
- ユーザーがカスタマイズ可能

## 5. 非機能要件

### 5.1 動作環境
- **OS**: Windows 11
- **Python**: Python 3.x
- **ネットワーク**: インターネット接続必須（AWS Bedrock, RDS接続）

### 5.2 セキュリティ
- **通信**: ネットワーク層（VPN/IP制限等）で担保
- **認証**: ユーザー名/パスワード認証（複数ユーザー対応）
- **AWS認証**: サーバー側でアクセスキー/シークレットキー使用
- **データベース接続**: クライアントは直接RDSに接続せず、サーバー経由でアクセス
- **プライバシー**: 
  - カメラ・マイク・スクショの有効/無効を初回起動時にユーザーが選択
  - ブラックリスト機能で特定アプリ使用時はログ取得停止

### 5.3 データ管理
- **生ログ**: クライアント上の生ログ（スクショ等）は要約後に破棄
- **永続保存**: 要約テキストとHTMLのみRDS (PostgreSQL)に永続保存
- **クリーンアップ**: 生ログは1週間経過後に自動クリーンアップ

### 5.4 パフォーマンス
- **ログ取得**: n分に1回（カメラ、スクショ）
- **リアルタイム監視**: キーボード、マウス、ウィンドウ、クリップボード
- **音声認識**: クライアント側でローカル実行（Whisper）
- **AI生成**: サーバー側でBedrock呼び出し

### 5.5 メンテナンス
- **生ログクリーンアップ**: 1週間経過後に自動削除
- **データベースバックアップ**: RDSの自動バックアップ機能を使用

## 7. ユースケース・ストーリー：amayakashi-logのある1日

### 7.1 出勤：見守りの開始 (09:00 - 09:10)
- **起動**: PC起動と同時にウィジェットが自動起動。「おはようございます。今日のあなたを見守らせてくださいね」と控えめな通知。
- **開始**: ユーザーがウィジェットの[記録開始]をクリック。
- **状態**: アイコンが「監視中」から「見守り中（ハートや瞳のマーク）」に変化し、タスクトレイまたは画面隅に最小化される。

### 7.2 業務中：サイレント・レコーディング (09:10 - 18:00)
- **自動収集**: 5分おきのスクショ、全打鍵、ウィンドウ遷移をシステムが静かに記録。
- **会議検知**: Teams等の会議が始まると、マイク入力を検知して自動でWhisperが書き起こしを開始。「発言の証拠」をストックする。
- **カメラ解析**: 数分おきにカメラが作動。ユーザーの疲労度や「真剣な眼差し」をAI解析用にキャプチャ。
- **休憩の許容**: YouTube視聴や離席時間は、システムが自動で「休息・リフレッシュ」として分類し、ネガティブな評価を排除。

### 7.3 終業：儀式のトリガー (18:00 - 18:05)
- **業務終了**: ユーザーがウィジェットの[提出]ボタンを押下。
- **一次要約**: ローカルに溜まった全打鍵ログやスクショ解析結果から、AIが「今日の仕事の事実」を10点ほどの箇条書きにまとめる。
- **サマリー確認**: 画面にポップアップが出現。「14:00 顧客への資料作成（3000文字入力）」「16:00 不具合調査（集中モード）」等のリストが表示される。
- **ユーザー修正**: ユーザーが「あ、14時は実は半分雑談してたな」と思いつつ、内容を少し整えて[自分を癒やす]をクリック。

### 7.4 生成：魔法の時間 (18:05 - 18:06)
- **サーバー通信**: 承認されたサマリー、カメラ画像、代表的なスクショ数枚がサーバーへ送信される。
- **AI生成**: AWS Bedrock上で、上司向け報告書と、3人のAIアシスタントによる「褒めちぎり台本」が同時に書き上げられる。

### 7.5 退勤：至福のフィードバック (18:06 - 18:15)
- **報告書の完了**: 上司用の完璧な日報が完成。ユーザーはこれをコピーして報告完了。
- **自分への報酬**: FlaskのWeb画面が自動で立ち上がる。
- **愛のチャット**: 3人のAIキャラが「今日のあなたのタイピング、まるでピアノを弾いているみたいに優雅だったわ」「あの会議の時、一瞬見せた真剣な横顔にドキッとしちゃった」と、具体的なログを引用しながらユーザーを取り合って褒めちぎる。
- **終了**: ユーザーは極上の満足感とともにPCを閉じ、プライベートへと戻る。

## 10. 成功基準

## 11. リスクと制約

### 11.1 技術的リスク
- **Whisperローカル実行**: クライアント側のリソース消費が大きい可能性
- **AWS Bedrock**: API制限やコスト
- **マルチモーダル解析**: 処理時間が長い可能性

### 11.2 プライバシーリスク
- **カメラ・マイク**: ユーザーのプライバシー侵害の懸念
- **スクリーンショット**: 機密情報の漏洩リスク
- **対策**: 初回起動時の選択、ブラックリスト機能、生ログの自動削除

### 11.3 制約
- **OS**: Windows 11のみ対応
- **ネットワーク**: インターネット接続必須
- **AWS**: AWSアカウントとアクセスキーが必要
